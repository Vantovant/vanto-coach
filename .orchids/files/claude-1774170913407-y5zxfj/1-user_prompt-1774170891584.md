MISSION FOR THIS SPRINT
We are now in Production Hardening Sprint 1.

The objective is:
Make failures visible, graceful, and recoverable.

This sprint is about observability, timeout protection, retry recovery, and regression safety.

IMPLEMENT IN THIS ORDER

1. ADD CENTRALIZED MONITORING
Implement a production-grade monitoring layer such as Sentry or Datadog.

Capture and report:
- client runtime errors
- server / edge failures
- Supabase request failures
- OpenAI / AI processing failures
- unexpected diary processing failures
- timeout-related failures

Important:
Include useful context in logs where possible, such as:
- authenticated user ID
- session ID
- operation type
- route or component context

2. REPLACE WEAK/BLOCKING ERROR HANDLING
Replace brittle browser alerts and weak error UX with clean, non-blocking toast notifications.

Use a polished toast system already compatible with the project style if available.

Create clear user-facing feedback for at least:
- failed diary save
- failed processing
- processing timeout
- failed delete
- retry started
- retry succeeded
- retry failed

3. ADD CLIENT-SIDE TIMEOUT PROTECTION
The UI must not allow a diary entry to remain stuck forever in a processing state.

Implement an artificial client-side timeout threshold for long-running diary processing.
If processing exceeds the allowed time window:
- force the UI state to stop waiting
- transition the entry from processing to failed
- display a clear failed state
- make the state recoverable

Do not rely only on the server to tell the client that it failed.

4. ADD RETRY PROCESSING
For any failed diary entry, add a visible “Retry processing” action.

Expected behavior:
- do not force the user to re-record audio
- do not discard the original saved diary record
- re-trigger the failed processing path using already-saved session data where possible
- preserve the original recording and transcript
- clearly show retry state in the UI

5. PROTECT THE EXISTING VERIFIED FLOW
After these changes, run a regression verification pass.

Re-check:
- record
- save
- refresh
- replay
- delete
- Today still reads real persisted data
- Memory still reads and persists real data
- no duplicate diary cards appear
- no destructive hard reload behavior is introduced

- do not weaken private storage assumptions
- do not introduce shortcuts that compromise private executive diary data
